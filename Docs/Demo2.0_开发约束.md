# Demo 2.0 开发约束

## 1. 版本目标

当前项目不是做完整长线游戏，而是做一个可持续扩展的战斗型 Demo。

当前阶段要验证的是：
- Boss 风格是否鲜明
- 掉落物是否足够有记忆点
- 不同职业与掉落物组合后，是否能形成明显的新兵种体验

不在当前阶段展开的内容：
- 角色成长
- 长线养成
- 商业化
- 大量数值平衡
- 大地图 / Roguelike 外层结构

## 2. 当前统一规格

- Unity 版本：`2022.3.62f1c1`
- 画面形式：`2D 横版像素`
- 战斗形式：`自动战斗`
- 玩家不手动移动
- 以关卡推进为主
- 以 Boss 掉落和职业组合为主循环

## 2.1 现有效果保护原则

当前已经做好的以下内容，默认视为可复用标准件，不轻易推翻重做：

- 第一关的捡锤子演出
- 三选一界面
- 发现新兵种演示

后续所有新关卡、新掉落物、新兵种演出，默认遵循：

- 优先复用现有结构与节奏
- 优先替换内容，不替换整套表现形式
- 如果必须新增步骤，优先插入现有链路前后，不拆掉原有骨架
- 新效果的黑底、发光、按钮风格、卡片结构、演示节奏，尽量与现有锤子版本保持一致

只有在原有效果无法承载新需求时，才允许做局部调整。

## 3. 战斗层固定结构

每个主关卡至少包含：
- 一段战斗内容
- 关卡目标明确的收束演出

其中：
- 第一、三、五关为 Boss 掉落关
- 第二、四、六关为验证 / 扩展 / 收束关

第一、三、五关固定包含：
- 一个 Boss
- 一个专属掉落物
- 一段拾取演出
- 一个三选一分配界面
- 一段发现新兵种演出

## 4. 文档分工

`demo2.0流程.txt`
- 负责定义关卡顺序
- 负责定义每关掉落什么
- 负责定义关卡之间如何衔接

`武器组合技能.txt`
- 负责定义每个掉落物交给不同职业后的结果
- 负责定义兵种能力概念
- 只要某个职业 / 物品组合已经出现在主线中，就默认以这份文档为唯一技能设计来源
- 后续详细设计与实现中，不再使用“待补”“还没编”作为主线技能口径

`主线组合总表.md`
- 负责把主线中真实会出现的职业 / 物品组合整理成二维表
- 负责给实现阶段提供快速查阅入口

`ResourceLibraries.md`
- 负责记录资源来源与当前资源库位置

## 5. 当前资源命名原则

玩家基础单位：
- `human_swordsman`
- `assassin`
- `longbowman`

第一层组合：
- `*_hammer`
- `*_fire`
- 后续如确认圣杯 / 大剑分支，再补对应命名后缀

Boss / 特殊敌人：
- 按资源名原义整理
- 例如 `goblin_boss_wife`
- 后续第二关、第三关 Boss 资源名待最终确定后再写入

## 6. 当前实现优先级

最高优先级：
- 六关流程顺序不被破坏
- 第一、三、五关的掉落闭环稳定
- 第二、四、六关的过场与验证职责清楚

中优先级：
- 第四关法师加入与换人流程
- 第六关最终验证与结尾镜头
- 新兵种的专属贴图替换
- 新投射物与特效表现

低优先级：
- 长线玩法文档
- 大量未来 Boss 的提前设计

## 7. 冗余控制原则

以后新增文档时遵循：
- 不重复写同一件事
- 战斗流程只写在流程文档里
- 组合能力只写在组合文档里
- 工程资源位置只写在资源文档里

如果某份文档已经被新文档覆盖，就删除，不保留平行版本。
